# EXECUTE_FLOW.md — Execute Page Architecture & AI Panel Reference

> **Purpose**: Documents the Execute page component tree, step data flow, Python code
> generation pipeline, run-log format, AI analysis integration, and the `AiAction` schema
> used for automated workflow fixes.

---

## 1. Execute Page Overview

The **Execute page** is where users run their automation workflows and review results.
It surfaces four key panels:

| Panel | Position | Purpose |
|-------|----------|--------|
| **StepListPanel** | Left | Ordered list of all steps; click to select |
| **CodePreviewPanel** | Center | Generated Python code (read-only) |
| **RunOutputPanel** | Right | Live execution log, findings, errors |
| **AiPanel** | Overlay / sidebar | AI-driven analysis and suggested fixes |

The page is rendered inside `ExecutePage` which receives the full application state
as props from `App.tsx`.

## 3. Step Data Flow

### 3.1 State Ownership

Steps are stored in `App.tsx` state as `Step[]` and passed down as props:

```typescript
interface Step {
  id: string;            // Unique step identifier
  type: StepType;        // 'connect' | 'write' | 'query' | 'sweep' | 'sleep' | 'python' | ...
  label: string;         // Human-readable description
  params: Record<string, any>;  // Type-specific parameters
  boundDeviceId?: string;       // Optional device binding
  children?: Step[];     // Nested steps (inside sweep/group)
}
```

### 3.2 Valid Step Types

| Step Type | Key Params | Notes |
|-----------|-----------|-------|
| `connect` | `instrumentIds: string[]`, `printIdn: boolean` | Opens device connections |
| `disconnect` | — | Closes all connections |
| `write` | `command: string` | SCPI write |
| `query` | `command: string`, `saveAs: string` (REQUIRED) | SCPI query |
| `set_and_query` | `command: string`, `queryCommand: string`, `saveAs: string` | Write then query |
| `sleep` | `duration: number` | Wait N seconds |
| `python` | `code: string` | Custom Python code |
| `save_waveform` | `source`, `filename`, `format` | |
| `save_screenshot` | `filename`, `scopeType: 'modern' \| 'legacy'` | |
| `recall` | `recallType`, `filePath`, `reference` | |
| `sweep` | `variableName`, `start`, `stop`, `step`, `saveResults` | Loop with children |
| `error_check` | `command`, `expectedValue` | |
| `comment` | `text` | |
| `group` | `children: Step[]` | Container with nested steps |

### Canonical Naming Rule

- Runtime `StepType` values in `src/App.tsx` are canonical (`write`, `query`, etc.).
- `scpi_write` and `scpi_query` appear as Blockly/internal block labels and conversion labels.
- For AI suggestions in Execute/Steps context, use runtime names:
  - `write` for SCPI write steps
  - `query` for SCPI query steps

### 3.3 Nested Steps

`sweep` and `group` steps can contain `children` arrays.  The generator recurses
into children to produce loop bodies or sequential groups.

Sweep loop generation in Python:
```python
# Sweep: variableName from start to stop step step
variableName = start
while variableName <= stop:
    # ... children commands ...
    variableName += step
```

## 4. Generated Python Code

### 4.1 Generation Functions

| Function | Backend | Description |
|----------|---------|-------------|
| `genSteps()` | Generic (PyVISA default) | Standard SCPI generation |
| `genStepsClassic()` | PyVISA | Classic PyVISA-only workflow |
| `genStepsTekHSI()` | TekHSI | gRPC-based generation |
| `genStepsVxi11()` | VXI-11 | Lightweight RPC generation |

The active function is selected based on the `connect_scope` block's `BACKEND` field,
NOT the UI configuration dropdown. The backend field in the XML/block is the source of truth.

### 4.2 Code Structure

Generated Python always follows this template:

```python
#!/usr/bin/env python3
"""Auto-generated by TekAutomate"""
import time
import pyvisa          # or: from tm_devices import DeviceManager

# --- Connection ---
rm = pyvisa.ResourceManager()
scope = rm.open_resource('TCPIP::192.168.1.100::INSTR')
smu = rm.open_resource('TCPIP::192.168.1.10::INSTR')

try:
    # --- Command Body ---
    scope.write('*RST')
    smu.write('OUTPut ON')

    v = 0.5
    while v <= 2.5:
        smu.write(f":SOURce:VOLTage {v}")
        scope.write('ACQuire:STATE OFF')     # acquisition_reset
        scope.write('ACQUIRE:STATE ON;*OPC?') # single_acquisition
        scope.read()
        freq = scope.query(':MEASUREMENT:IMMED:VALUE?').strip()
        v += 0.5

finally:
    # --- Cleanup ---
    if 'scope' in locals():
        scope.close()
        print("Disconnected scope")
    if 'smu' in locals():
        smu.close()
        print("Disconnected smu")
```

### 4.3 Multi-Device Code Generation

- Each device gets its own named variable (`scope`, `psu`, `smu`, `dmm`, etc.)
- Commands reference the correct variable based on `DEVICE_CONTEXT` field
- Cleanup closes **all** opened devices symmetrically
- Device names are extracted from generated code via regex for reliability

### 4.4 Backend-Specific Generation

| Backend | Connection Code | Command Code | Cleanup |
|---------|----------------|-------------|--------|
| PyVISA | `rm.open_resource(...)` | `device.write(...)` / `.query(...)` | `device.close()` |
| tm_devices | `DeviceManager().add_scope(...)` | `device.commands.acquire.state.write(...)` | `dm.close()` |
| TekHSI | `tekhsi.connect(...)` | `scope.get_data("CH1")` | `scope.close()` |
| Hybrid | Both PyVISA + TekHSI connections | SCPI → PyVISA, waveforms → TekHSI | Close both |
| VXI-11 | `vxi11.Instrument(...)` | `device.write(...)` / `.ask(...)` | `device.close()` |

## 5. Run Log Format

During execution, the system produces a sequence of log entries:

```typescript
interface RunLogEntry {
  timestamp: string;    // ISO 8601
  stepId: string;       // Which step produced this entry
  status: 'success' | 'error' | 'warning' | 'info';
  message: string;      // Human-readable summary
  data?: {
    command?: string;       // SCPI command that was sent
    response?: string;      // Query response (if applicable)
    exceptionType?: string; // Python exception class
    traceback?: string;     // Full traceback
    duration?: number;      // Execution time in ms
  };
}
```

### 5.1 Error Entries

When a step fails, the log entry includes:
- `exceptionType`: e.g., `visa.errors.VisaIOError`, `TimeoutError`
- `traceback`: Full Python traceback string
- `command`: The SCPI command that triggered the error
- `status`: Always `'error'`

### 5.2 Log as AI Input

The run log is the primary input for AI analysis. The AI panel assembles:
1. Full run log (all entries)
2. Current flow JSON (step definitions + parameters)
3. Selected step context (if user has clicked a specific step)

## 6. Selected Step Tracking

- `selectedStep` state tracks which step the user has clicked in `StepListPanel`
- Clicking a step highlights it and scrolls `CodePreviewPanel` to the corresponding line
- The AI panel uses `selectedStep` for context-aware suggestions:
  - If a step is selected, findings related to that step are prioritised
  - The step editor panel shows editable params for the selected step
- Selection can be cleared by clicking elsewhere or pressing Escape

## 7. AI Analysis Flow

### 7.1 Trigger

User clicks **"Analyze Current Run"** button in the AI panel.

### 7.2 Assembly

The system assembles a context payload:

```typescript
interface AiAnalysisPayload {
  runLog: RunLogEntry[];       // Full execution log
  flowJson: Step[];            // Current workflow definition
  selectedStepId?: string;     // Currently selected step (optional)
  generatedCode: string;       // Python code that was executed
  deviceConfig: DeviceEntry[]; // Device connection details
}
```

### 7.3 Analysis

Currently, the payload is formatted and copied for use with an external AI (e.g., GPT).
Future: direct API call to AI backend.

### 7.4 Response

The AI returns two arrays:
- **Findings**: Problems identified in the run
- **Actions**: Suggested fixes to apply

## 8. AiFinding & AiAction Schema

### 8.1 AiFinding

```typescript
interface AiFinding {
  id: string;                   // Unique finding identifier
  severity: 'critical' | 'warning' | 'info';
  category:
    | 'scpi_error'              // SCPI command failed
    | 'timing'                  // Timing/timeout issue
    | 'device_mismatch'         // Wrong device context
    | 'missing_step'            // Required step absent
    | 'code_quality'            // Code improvement suggestion
    | 'backend_incompatibility' // Wrong block for backend
    | 'variable_issue';         // Undefined/unused variable
  symptom: string;              // One-line description of what went wrong
  affectedStepId?: string;      // Step that caused the issue
  confidence: number;           // 0.0–1.0 confidence score
  details: string;              // Full explanation with context
}
```

### 8.2 AiAction

```typescript
interface AiAction {
  id: string;                   // Unique action identifier
  findingId: string;            // Which finding this action addresses
  type:
    | 'modify_step'             // Update params of existing step
    | 'add_step'                // Insert new step at position
    | 'remove_step'             // Remove step by id
    | 'reorder'                 // Move step to new position
    | 'change_param';           // Update specific param within step
  targetStepId?: string;        // Step to modify (for modify/remove/change_param)
  insertAfterStepId?: string;   // Insertion point (for add_step)
  description: string;          // Human-readable explanation of the change
  changes: Record<string, any>; // Key-value pairs of param changes
}
```

### 8.3 Example Finding + Action Pair

```json
{
  "finding": {
    "id": "f1",
    "severity": "critical",
    "category": "device_mismatch",
    "symptom": "Measurement command sent to SMU instead of scope",
    "affectedStepId": "step-42",
    "confidence": 0.95,
    "details": ":MEASUREMENT:IMMED:TYPE FREQUENCY is a scope-specific command but the step targets device 'smu'. This will cause a VISA error."
  },
  "action": {
    "id": "a1",
    "findingId": "f1",
    "type": "change_param",
    "targetStepId": "step-42",
    "description": "Change device context from (smu) to (scope)",
    "changes": { "boundDeviceId": "scope-device-id" }
  }
}
```

## 9. `onApplyAiActions` Function

Applies an array of `AiAction` to the current flow state.

### 9.1 Action Processing

```typescript
function onApplyAiActions(actions: AiAction[]): void {
  let updatedSteps = [...currentSteps];

  for (const action of actions) {
    switch (action.type) {
      case 'modify_step':
        // Find step by targetStepId, merge action.changes into step.params
        updatedSteps = updateStepParams(updatedSteps, action.targetStepId, action.changes);
        break;

      case 'add_step':
        // Create new step from action.changes, insert after insertAfterStepId
        updatedSteps = insertStep(updatedSteps, action.insertAfterStepId, action.changes);
        break;

      case 'remove_step':
        // Remove step by targetStepId
        updatedSteps = removeStep(updatedSteps, action.targetStepId);
        break;

      case 'reorder':
        // Move step to new position specified in action.changes.newIndex
        updatedSteps = moveStep(updatedSteps, action.targetStepId, action.changes.newIndex);
        break;

      case 'change_param':
        // Update specific param key within step
        updatedSteps = changeParam(updatedSteps, action.targetStepId, action.changes);
        break;
    }
  }

  setSteps(updatedSteps);
  regenerateCode(updatedSteps);
}
```

### 9.2 Nested Step Handling

For steps inside `sweep` or `group` containers, the update functions recurse into
`children` arrays to find the target step by ID.

## 10. Audit Trail

Every AI action application is logged for traceability:

```typescript
interface AuditEntry {
  timestamp: string;           // When the action was applied
  actionId: string;            // Which AiAction was applied
  findingId: string;           // Which finding it addressed
  beforeState: Step[];         // Snapshot before change
  afterState: Step[];          // Snapshot after change
  appliedBy: 'user' | 'auto'; // Who triggered the apply
}
```

### 10.1 Undo Support

- Each audit entry preserves `beforeState` so changes can be reverted
- User can click "Undo" on any applied action to restore previous state
- Undo replays the `beforeState` snapshot and regenerates Python code

### 10.2 Audit Log Display

The AI panel shows a history of applied actions with:
- Timestamp
- Finding severity badge
- Action description
- Undo button

## 11. Bidirectional Sync with Blockly

The Execute page works with both the Steps UI and the Blockly Builder:

### 11.1 Steps → Blockly

| Steps Step | Blockly Block | Notes |
|-----------|--------------|-------|
| `connect` | `connect_scope` | Looks up device config for backend/IP |
| `write` | `scpi_write` | |
| `query` | `scpi_query` | `saveAs` → `VARIABLE` |
| `sweep` | `controls_for` | `variableName/start/stop/step` → `VAR/FROM/TO/BY` |
| `sleep` | `wait_seconds` | |
| `python` | `python_code` | |
| `group` | (flattened) | Children extracted sequentially |

### 11.2 Blockly → Steps

| Blockly Block | Steps Step | Notes |
|--------------|-----------|-------|
| `connect_scope` | `connect` | |
| `scpi_write` | `write` | |
| `scpi_query` | `query` | |
| `controls_for` | `sweep` | `params.variableName`, `params.start`, `params.stop`, `params.step` |
| `controls_repeat_ext` | `sweep` | `params.variableName='i'`, computed start/stop/step |
| `wait_seconds` | `sleep` | |
| `python_code` | `python` | |
| `variables_set` | `python` | Variable assignment as Python |
| `set_device_context` | `comment` | Device switch recorded |

### 11.3 Round-Trip Fidelity

The conversion is tested via E2E tests (`e2e/flow-fidelity.spec.ts`,
`e2e/set-and-query-canary.spec.ts`) that verify Steps → Blockly → Steps produces
equivalent output. Known limitations:
- Complex nested conditions lose structure (converted to `python` steps)
- Device context switches become `comment` steps
- Advanced math expressions are serialised as Python code

## 12. Flow Designer (Advanced Mode)

The Flow Designer provides a graph-based alternative to linear steps:

### 12.1 Node Types

| Node Type | Purpose | Key Properties |
|-----------|---------|----------------|
| `trigger` | Flow entry point | `type: 'manual' \| 'schedule' \| 'event'` |
| `group` | Sequential command container | `instrumentId`, `instrumentAlias`, `commands[]` |
| `condition` | If/else branching | `conditionExpression`, `nextTrue`, `nextFalse` |
| `loop` | Iteration | `loopType`, `loopExpression`, `next` |
| `delay` | Wait | `waitTime` (seconds) |
| `verify` | Retry with check | `verifyCmd`, `expectedResponse`, `retryCount` |
| `python` | Custom code | `pythonCode`, `passCriteria`, `failCriteria` |
| `terminate` | Flow exit | — |

### 12.2 Graph Traversal for Code Generation

The Python generator traverses the flow graph depth-first:

```typescript
function traverse(nodeId: string): void {
  const node = nodeMap.get(nodeId);
  if (!node || visited.has(nodeId)) return;
  visited.add(nodeId);

  if (node.type === 'condition') {
    generateCondition(node);
    traverse(node.nextTrue);
    generateElse();
    traverse(node.nextFalse);
  } else if (node.type === 'loop') {
    generateLoop(node);
    traverse(node.next);
    closeLoop();
  } else {
    generateNode(node);
    traverse(node.next);
  }
}
```

### 12.3 Instrument Binding

- Each node can specify `instrumentId` or `instrumentAlias`
- Commands within groups can override the group's instrument
- Resolution: command-level → group-level → default device

## 14. Error Handling & Validation

### 14.1 Pre-Generation Validation

Before generating Python, the system validates:
- Backend compatibility (no `scpi_write` with `tm_devices`)
- Device IP conflict detection (no duplicate IPs)
- Command-to-device mapping (scope commands → scope devices)
- Variable usage (all assigned variables should be used)
- Device usage (all connected devices should have operations)

### 14.2 Runtime Error Handling

Generated Python wraps execution in `try/finally`:
```python
try:
    # ... workflow commands ...
except Exception as e:
    print(f"Error: {e}")
    raise
finally:
    # Cleanup: close all devices
    for device_name in ['scope', 'smu', 'psu']:
        if device_name in locals():
            locals()[device_name].close()
```

### 14.3 Validation Error Display

Validation errors appear as banners at the top of the Execute page with:
- Error type and severity
- Affected block/step identification
- "HOW TO FIX" guidance
- Link to relevant documentation
